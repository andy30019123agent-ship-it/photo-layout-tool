## 1. BUG（實際與潛在、邊界、大圖記憶體）

### 🔴 大圖記憶體爆炸（已知問題，尚未修）

**問題：** 沒有張數上限、也沒有單檔大小上限。以 A4 300dpi 計算，單張 canvas 需要 ~33 MB 記憶體；匯出 50 張照片（13 頁）需建立 13 個大 canvas，共 ~431 MB。加上每張 4MB dataUrl 的 50 張照片本身就達 200MB，總計輕易超過 600 MB，行動裝置 Safari 會直接閃退。

**怎麼修：**
1. 在 `UploadZone.jsx` 的 `handleFiles` 函式加上累計張數限制（建議最多 30 張）及單檔大小上限（建議 20 MB）：
   ```js
   const MAX_FILES = 30;
   const MAX_SIZE_MB = 20;
   // 過濾超大檔
   const valid = Array.from(files)
     .filter(f => {
       if (f.size > MAX_SIZE_MB * 1024 * 1024) {
         onError(`${f.name} 超過 ${MAX_SIZE_MB}MB，已略過`);
         return false;
       }
       return ACCEPTED_EXTS.includes('.' + f.name.split('.').pop().toLowerCase()) || f.type.startsWith('image/');
     })
     .slice(0, MAX_FILES - currentCount);
   ```
2. 在 `exportEngine.js` 的 `renderFullPage` 使用完後，立即釋放 canvas：
   ```js
   // 用完後清零，讓 GPU 記憶體可以回收
   canvas.width = 0;
   canvas.height = 0;
   ```

---

### 🟡 `getPaperDimensions` 找不到紙張時會 crash

**問題：** `canvasEngine.js` 第 10–11 行，`PAPER_SIZES.find()` 找不到時回傳 `undefined`，接著解構 `{ widthMm, heightMm }` 會噴 `TypeError: Cannot destructure property 'widthMm' of 'paper' as it is undefined`，整個畫面白屏。

**發生時機：** 理論上 UI 只提供合法值，但若未來新增/移除 paper 選項而 DEFAULT_SETTINGS 沒同步更新，就會爆。

**怎麼修：** 加 fallback：
```js
const paper = PAPER_SIZES.find(p => p.id === settings.paperSize) ?? PAPER_SIZES[0];
```

---

### 🟡 canvas 匯出後未釋放記憶體

**問題：** `exportEngine.js` 的 `exportPDF` 和 `exportImages` 每個迴圈都建立 `canvas` 但結束後沒有清除（`canvas.width = 0`），在頁數多時會造成瀏覽器 GPU 記憶體洩漏，最終導致匯出失敗或標籤頁崩潰。

**怎麼修：** 在 `canvasToBlob` 呼叫後、進入下一頁前加上：
```js
canvas.width = 0; canvas.height = 0;
```

## 2. 安全性

### 🟢 無金鑰外洩風險

純前端工具，沒有 API Key、Token、後端連線。掃描 `src/`、`index.html`、`vite.config.js` 均未發現任何密碼或金鑰字眼。**此項安全。**

---

### 🟡 檔案類型驗證有繞過空間

**問題：** `UploadZone.jsx` 的 `handleFiles` 允許 `f.type.startsWith('image/')` 的所有類型，攻擊者可以把 SVG 偽裝成 image/svg+xml 上傳。SVG 可以嵌入 `<script>` 標籤，但因為最終是透過 `createImageBitmap → canvas → dataUrl` 轉換，XSS 風險已被隱性消除。

**怎麼修（加固）：** 建議在 `isAccepted` 只允許白名單 MIME：
```js
const ACCEPTED_TYPES = ['image/jpeg', 'image/png', 'image/webp', 'image/heic', 'image/heif'];
function isAccepted(file) {
  const ext = '.' + file.name.split('.').pop().toLowerCase();
  return ACCEPTED_EXTS.includes(ext) && ACCEPTED_TYPES.includes(file.type);
}
```
注意：`ACCEPTED_TYPES` 本來在 `constants.js` 有定義但從未被引用（已於本次盤點刪除，見第4項），此修正需重新加回並引用。

---

### 🟢 無 XSS 風險

錯誤訊息以純文字 state 儲存並透過 React JSX 渲染，不走 `dangerouslySetInnerHTML`，照片名稱也僅用於顯示 `alt` 和 `title`。**此項安全。**

## 3. 成本與風控

### 🟢 純前端零後端成本

完全在瀏覽器端處理，無任何 API 呼叫、伺服器費用、外部服務依賴。就算 100 人同時使用，成本仍是 0 元（GitHub Pages 免費）。**此項安全。**

---

### 🟡 匯出失敗沒有部分成功提示

**問題：** `exportPDF` / `exportImages` 若中途某頁 `renderFullPage` 或 `canvas.toBlob` 失敗，會直接 throw 整個流程中斷。用戶只看到頂部紅色錯誤條，但不知道已成功幾頁。

**怎麼修：** 在 `exportImages` 的 zip 迴圈改成 try/catch per page，失敗的頁標記後繼續：
```js
for (let i = 0; i < totalPages; i++) {
  try {
    const canvas = renderFullPage(photos, settings, i);
    const blob = await canvasToBlob(canvas, format, quality);
    zip.file(`${base}_p${i + 1}.${format}`, blob);
  } catch (err) {
    onProgress(`第 ${i + 1} 頁失敗，跳過`);
  }
}
```

## 5. 邏輯衝突

### 🟢 `photosPerPage === 0` 的 getGridLayout 邊界

**問題：** `getGridLayout(0, ...)` 會回傳 `{ cols: 0, rows: null }`，導致後續 `cols * rows` 計算迴圈次數為 0（不崩潰但什麼也不畫），且 `availW / 0` 會得到 `Infinity` 的 cellW。

**影響：** 目前 UI 的 LAYOUT_OPTIONS 只提供 4 和 6，photosPerPage 永遠不會是 0，所以實際上不會觸發。但若日後擴充選項需注意。

**怎麼修（防禦性）：** 在 `getGridLayout` 加入 guard：
```js
if (!photosPerPage || photosPerPage <= 0) return { cols: 1, rows: 1 };
```

## 7. 相依套件風險（npm audit）

### 🔴 4 個漏洞（含 1 critical、1 high、2 moderate）

```
dompurify <=3.4.10   → XSS、prototype pollution 等 13 個 CVE
jspdf <=4.2.0        → 依賴有漏洞的 dompurify（optional dependency）
esbuild <=0.24.2     → dev server 任意請求讀取（僅影響開發期）
vite <=6.4.2         → 依賴有漏洞的 esbuild
```

**風險評估：**  
- `dompurify` 是 jspdf 的 **optional** dependency，本專案未直接呼叫 jspdf 的 HTML 渲染功能（只用 `addImage`），因此 dompurify 的 XSS 漏洞**在本專案不會被觸發**。但 dist/assets 中確實有打包進 `purify.es-BwoZCkIS.js`（非必要）。  
- `esbuild` 漏洞只在本機 `npm run dev` 時有風險，線上版本不受影響。

**怎麼修：**
```bash
# 升級 vite（breaking change，需測試）
npm install --save-dev vite@latest

# jspdf 升到 4.x 可解 dompurify（但 API 有 breaking change，需測試匯出功能）
npm install jspdf@latest
```
建議：先在本機 `npm run dev` 測試功能正常後，再 `npm run build` 驗證 dist，確認後再部署。

## 9. Git 衛生（重點）

### 🔴 `node_modules/` 和 `dist/` 被追蹤在 git 中（7,151 個檔案）

**問題：**
- `node_modules/`（7,141 個追蹤檔案）和 `dist/`（10 個追蹤檔案）都已進入 git index。
- `.gitignore` 已正確列出 `node_modules` 和 `dist`，但因為這兩個目錄在 .gitignore 加入前就已被 commit，所以 `.gitignore` 對它們無效。
- 每次 `npm install` 或 `npm run build` 後，`git status` 都會顯示大量雜訊，干擾真正的程式碼異動追蹤。
- Repo 體積膨脹，push/clone 極慢。

**怎麼修（需要 Andy 執行）：**
```bash
# 步驟 1：從 git index 移除 node_modules 和 dist（不刪除本機檔案）
git rm -r --cached node_modules/
git rm -r --cached dist/

# 步驟 2：確認 .gitignore 已有正確內容（現在已有，不需改）

# 步驟 3：commit 這個「清理 git 追蹤」的變更
git commit -m "chore: stop tracking node_modules and dist"

# 步驟 4：push
git push
```

> ⚠️ 上述指令會讓 GitHub 上的 repo 歷史仍保留舊的 node_modules，但之後不會再追蹤新的異動。若要完全清除歷史中的 node_modules（大幅減小 repo），需要用 `git filter-branch` 或 `git-filter-repo`，但風險較高，非必要不建議。

## 10. 優化建議

### 🟢 加入張數/大小上限提示 UI

目前超過限制只會在 error-bar 顯示，建議在 UploadZone 的說明文字也加上「最多 XX 張，單檔上限 XX MB」，讓用戶在上傳前就知道限制。

---

### 🟢 考慮 DPI 選擇 UI

目前 DPI 硬編碼在 `DEFAULT_SETTINGS.dpi = 300`，沒有 UI 可調整。若用戶想要更輕量的輸出（如 150dpi），需要修改程式碼。建議在 SettingsPanel 加入 DPI 選擇（150/200/300），同時告知不同 DPI 對應的輸出檔案大小。

---

### 🟢 HEIC 轉換進度提示

HEIC 轉 JPEG 可能需要 1–5 秒（依圖片大小），目前 `isLoading` 的 「載入中...」提示只是全域 loading，沒有逐張進度（如「正在轉換 2/5 張...」）。改善體驗建議：在 `handleFilesAdded` 迴圈中加入 per-file 進度狀態。

---

### 🟢 `jspdf` optional dependencies 可能打包進 dist

從 `dist/assets/` 可見 `purify.es-BwoZCkIS.js` 和 `html2canvas.esm-CBrSDip1.js` 被打包進去，但本專案完全沒用到這些功能。升到 jspdf 4.x 或在 vite.config.js 加入 `optimizeDeps.exclude` 可以減少 bundle 大小：
```js
optimizeDeps: {
  exclude: ['dompurify', 'html2canvas', 'canvg'],
}
```
